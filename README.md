# Hñäki-app (Frontend Client)
### Geospatial Intelligence Interface for Agricultural Decision Making.
*The public-facing client for the Tepozixtli asynchronous engine.*

## Bridging the Data Gap
While the backend engine (Tepozixtli) handles the heavy lifting of satellite telemetry ingestion and geospatial mathematics, Hñäki-app is tasked with the ultimate goal of the project: delivering actionable intelligence directly into the hands of agricultural decision-makers.

The challenge in geospatial web development is rendering massive amounts of data (like Cloud Optimized GeoTIFFs derived from Copernicus Sentinel-1/2) without overwhelming the user's browser or device memory.
## The User Experience (UX) Focus
Hñäki-app is designed as a highly optimized Single Page Application (SPA). It provides a seamless, cinematic map interface where producers and analysts can select specific municipalities or draw custom Areas of Interest (AOIs). Instead of forcing the user to download gigabytes of raster data, the application intelligently requests only the specific pixels needed for the current viewport, ensuring lightning-fast rendering even on constrained network connections.

## Rendering Architecture: The Geospatial Challenge
Building a web client capable of displaying high-resolution satellite imagery requires bypassing the standard DOM and rendering directly on the GPU. Here is how Hñäki-app achieves 60fps cinematic map interactions:

### The WebGL Engine (MapLibre GL JS)
To maintain vendor neutrality and adhere strictly to open-source principles, the core map engine is built on **MapLibre GL JS** (interfaced via `react-map-gl`). This WebGL-powered canvas ensures that complex vector boundaries and dense raster pixels are processed directly by the device's graphics card, completely freeing the browser's main thread. Custom Area of Interest (AOI) creation is seamlessly injected into the canvas using `@mapbox/mapbox-gl-draw`.

### Smart Raster Rendering (HTTP Range Requests)
The heaviest assets in this ecosystem are the satellite rasters generated by the Tepozixtli engine. Instead of forcing the client to download a massive standard TIFF file, the frontend leverages the Cloud Optimized GeoTIFF (COG) format alongside `geotiff.js`.
When a user navigates the map, the `RasterLayerManager` executes HTTP Range Requests. It calculates the current bounding box and zoom level, and downloads only the specific byte-blocks of pixels visible on the screen. This translates to instant visual feedback and prevents memory exhaustion on lower-end devices.}

### Dynamic Vector Wrapping
Geometries (such as municipal borders) are retrieved as raw PostGIS outputs from the backend. Rather than burdening the server with format conversion overhead, the React client dynamically wraps these raw geometries into standard `GeoJSON` Feature collections on the fly, feeding them directly into the MapLibre source layers.

## State Management & Network Architecture
Handling complex geospatial operations and asynchronous data fetching in a React application can easily lead to "prop-drilling" and bloated state management. Hñäki-app avoids this by strictly separating server state from client state:

### Server State (TanStack Query)
We completely abandoned traditional `useEffect` data fetching. All communication with the FastAPI backend is orchestrated through TanStack Query (React Query). This provides out-of-the-box caching, request deduplication, and automatic background revalidation. When a new satellite processing task is completed, React Query intuitively invalidates the specific cache keys, ensuring the map updates organically without full page reloads.

### Network Layer & Edge Proxying
Data fetching is standardized through a custom `fetch` API wrapper (`apiClient.ts`), guaranteeing consistent error handling and payload parsing across all modules.
Crucially, to securely communicate with the DigitalOcean backend and avoid browser-level CORS (Cross-Origin Resource Sharing) or "Mixed Content" blocks (HTTPS to HTTP), the application utilizes a Reverse Proxy configured at the edge via `netlify.toml`. The React client makes clean, relative requests (e.g., `/api/v1/...`), which Netlify transparently intercepts and routes to the production Droplet.

### Client State (Minimalist Approach)
By delegating all remote data management to TanStack Query, the global client state footprint is drastically reduced. We deliberately avoided heavy boilerplate libraries like Redux. UI-specific state (such as the currently active municipality or selected raster layer) is managed via localized `useState` hooks, keeping the component tree lightweight and highly performant.

## Engineering Decisions & Trade-offs
To ensure the frontend remains as agile as the asynchronous backend, we made strict choices regarding our build tools, styling, and file architecture:

### Build Tooling: Vite over CRA

* **The Rationale:** Traditional bundlers (like Webpack via Create React App) struggle with cold start times and slow Hot Module Replacement (HMR) as the project scales.
* **The Solution:** We adopted Vite as our build tool. By leveraging native ES modules in the browser, Vite provides near-instantaneous server starts and lightning-fast HMR. This is critical when iteratively styling complex WebGL map layers, where waiting for a rebuild breaks the developer flow.

### Architecture: Feature-Sliced Design (FSD) Adaptation
* **The Rationale:** Grouping files by technical type (e.g., putting all hooks in a single `/hooks` folder and all components in `/components`) creates deeply coupled, hard-to-navigate codebases as the app grows.
* **The Solution:** The `src/features/` directory implements a relaxed Feature-Sliced Design. Code is grouped by business domain (e.g., `municipios`, `indicadores-macro`). Each feature is treated as a self-contained module with its own `api`, `ui`, `model`, and `lib`.
* **The Trade-off:** It requires more discipline to avoid cross-feature imports, but it guarantees that deleting or refactoring a feature (like we did with legacy modules) has zero domino effect on the rest of the application.

### Styling: Utility-First (Tailwind CSS)
* **The Rationale:** Geospatial applications often suffer from CSS bloat due to complex overlapping UI controls (floating panels, map legends, navigation).
* **The Solution:** We utilize Tailwind CSS. As seen in our `MainLayout` and floating map controls, UI elements are styled directly via utility classes. This eliminates context-switching between CSS and TSX files and ensures the final production CSS bundle remains exceptionally small, balancing the heavy weight of our WebGL dependencies.

## Development Philosophy & Traceability
Just like its backend counterpart, this repository is governed by strict engineering discipline. The Git history is not an afterthought; it is a core component of the project's documentation.

### Atomic Commits & Issue-Driven Development
Every UI component, custom hook, or API integration is backed by a specific GitHub Issue. We enforce atomic commits—meaning each commit represents a single, logical change. This guarantees that any developer can trace a visual change in the frontend directly back to its original business requirement, making code reviews highly efficient and potential rollbacks painless.

## Future Roadmap: Beyond the Hackathon
Hñäki-app was originally conceptualized during the HackatonLAC Copernicus 2026. While the current production version prioritizes speed, UX, and public data delivery, the architecture is primed for its next evolutionary phase:

* **Admin Layer & RBAC:** We are planning the re-integration of a Role-Based Access Control (RBAC) system. This will activate our isolated AdminLayout to securely manage data ingestion workflows and trigger satellite processing tasks directly from the UI.

* **Global Client State (Zustand):** As the Admin layer introduces more complex cross-component state requirements (e.g., user sessions, granular permissions), we will leverage Zustand to handle the global client state cleanly, preserving TanStack Query exclusively for server-state synchronization.
